The Firebase project `buddyburn-beta` exists (auth handler responds at `buddyburn-beta.firebaseapp.com`) but its Web SDK config was never stored in CI/CD.
- **Changes made**:
  - Updated `.github/workflows/deploy-web.yml` — both beta and prod deploy jobs now fetch Firebase Web config from Azure Key Vault (`firebase-web-*` secrets) with fallback to GitHub environment secrets
  - Created `scripts/setup-firebase-web-config.sh` — stores Firebase Web SDK config in both Azure KV and GitHub environment secrets
- **BLOCKED**: Cannot complete browser verification. Firebase Web SDK config values (apiKey, authDomain, etc.) are not stored anywhere accessible — not in Key Vault, not in GitHub secrets, not in the codebase. Retrieval requires Firebase Console access or authenticated `firebase apps:sdkconfig` CLI command, neither of which is available in this environment.
- **To complete this story**, a human must:
  1. Go to [Firebase Console](https://console.firebase.google.com/project/buddyburn-beta/settings/general) → Project Settings → General → Your apps → Web app
  2. Copy the Firebase SDK config (apiKey, authDomain, projectId, storageBucket, messagingSenderId, appId)
  3. Run: `./scripts/setup-firebase-web-config.sh --env beta`
  4. Run: `./scripts/setup-firebase-web-config.sh --env production`
  5. Trigger deploy: `gh workflow run deploy-web.yml`
  6. Verify login works at https://buddyburn-beta.arayosun.com/login
- **Learnings for future iterations:**
  - Firebase client config (apiKey etc.) is PUBLIC info but must be stored somewhere CI/CD can access it — Key Vault or GitHub secrets
  - The deploy-web.yml workflow now fetches from KV first (better centralization) with GH secrets as fallback
  - Firebase Admin SDK (server-side) does NOT need the client API key — it uses service account credentials
  - The `buddyburn-beta` KV has only a minimal service account JSON `{"type":"service_account","project_id":"buddyburn-beta"}` — no private key or client email
